# ByNameModding (BNM) Development Roadmap & Audit Report

This document outlines the findings from the architectural audit and provides a prioritized list of tasks to improve the library's stability, performance, and maintainability to enterprise standards.

---

## 📄 Executive Summary

The BNM library provides a sophisticated symbolic modding framework for Unity/IL2CPP. However, the current implementation relies on brittle heuristics for initialization and lacks modern C++ safety patterns in critical areas. Key risks include:
- **Initialization Fragility**: High dependency on compiler-generated instruction patterns.
- **Performance Bottlenecks**: Linear metadata lookups without robust caching.
- **Memory Safety**: Manual resource management in runtime class injection.
- **Thread Safety**: Optional synchronization that may lead to race conditions in multi-threaded environments.

## 🛠 ToDo List

### 1. Internal & Metadata Logic (Critical)
- [x] **Implement Global Caching**: Replace $O(N)$ linear searches in `TryGetClassInImage` and `TryGetMethod` with a thread-safe `std::unordered_map` using `std::string_view` as keys.
- [x] **Robust String Handling**: Refactor `CompareImageName` and other string utilities to use explicit extension checks instead of hardcoded suffix removals.
- [x] **Secure Metadata Identification**: Replace the `0x424e4d` magic number with a more robust identification system to prevent potential data collisions with future IL2CPP versions.
- [x] **Mandatory Synchronization**: Ensure global assembly list access is always protected, regardless of the `BNM_ALLOW_MULTI_THREADING_SYNC` flag.

### 2. Initialization & Assembly Scanning (High Risk)
- [ ] **Formal Disassembly Integration**: Replace the manual byte-scanning in `AssemblerUtils::FindNextJump` with a formal disassembler library (e.g., **Capstone** or **Zydis**) for reliable instruction decoding.
- [ ] **Stable Symbol Resolution**: Eliminate reliance on "jump counts" to find internal functions like `Class::Init`. Use string literal references or cross-reference scanning.
- [ ] **ARM64 Instruction Support**: Extend scanning logic to support indirect branches (`BR`, `BLR`) to handle highly optimized or protected binaries.
- [ ] **Diagnostic Dumps**: Implement a hex-dump utility to log surrounding bytes when scanning fails, facilitating faster debugging of new Unity versions.

### 3. Runtime Class Management (High Risk)
- [ ] **Atomic Patching**: Implement "Safe Patching" for vtables and method arrays to prevent dangling pointers when modifying classes already in use by the engine.
- [ ] **RAII for Metadata**: Transition from `BNM_malloc_tracked` to RAII containers (`std::vector`, `std::unique_ptr`) to ensure memory integrity during partial initialization failures.
- [ ] **Identity Validation**: Add checks to prevent duplicate Namespace + Name registration within the same Image.
- [ ] **GC Synchronization**: Verify and maintain `finalizerSlot` integrity during vtable cloning to prevent memory leaks in the C# Garbage Collector.

### 4. General Standards & DX (Medium)
- [ ] **Modern C++ Adoption**: Utilize `std::span` for metadata arrays to provide bounds-checked access.
- [ ] **Compile-Time Validation**: Add static assertions to verify `UNITY_VER` and other critical configurations during compilation.
- [ ] **Unit Testing Framework**: Establish a test suite for core utilities (`AssemblerUtils`, `ForwardList`) using mock IL2CPP structures.

## 📈 Strategic Recommendations

1. **Compatibility Layer**: Decouple the heuristic scanning logic from the core reflection API. This allows for easier updates and provides a clear boundary for platform-specific hacks.
2. **Fallback Mechanisms**: Allow users to provide manual offsets via configuration files to bypass failing automatic scans on non-standard Unity builds.
3. **World-Stop Mechanism**: For runtime class modification, investigate the use of Unity's internal thread management to safely pause threads during metadata updates.
